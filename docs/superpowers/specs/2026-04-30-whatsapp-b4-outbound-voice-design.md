# WhatsApp B4 — Outbound Voice Notes — Design

**Date:** 2026-04-30
**Status:** Approved (in brainstorm), pending spec review and user review.
**Scope:** Sub-project B4. Last sub-project of the WhatsApp roadmap.
**Depends on:** sub-A (`sendOutboundText` opts pattern, mirrored here for `sendOutboundVoice`), B0 (agent loop pattern), B3a (no-tool-uses branch shape), B3b (`latestReportPdf` precedence + branch ordering).

---

## 1. Purpose

When the owner sends a **voice note** (`inbound.tipo === 'audio'`) and the agent's reply text is short (≤400 chars), Maia answers with a **voice note** generated by OpenAI TTS (`tts-1`, voice `'nova'`, OGG-Opus output) and delivered via Baileys with `ptt: true` so it renders as a voice-note bubble (not a music-player attachment).

For everything else — text-inbound conversations, replies longer than 400 chars, turns that produced a PDF, briefings/proactive messages, alerts — the existing text path remains unchanged. Voice is strictly a **symmetric reply channel**: voice-in implies voice-out, nothing more.

This makes voice-note conversations feel conversational ("registra R$ 500 do Uber" → "✅ R$ 500 registrado em Transporte") instead of asymmetrically text-replying to spoken input.

## 2. Goals

- Owner sends voice → Maia replies voice (≤400 chars). Owner sends text → Maia replies text. Owner sends voice on a long-reply turn → Maia falls back to text silently.
- New `sendOutboundVoice` in `baileys.ts`; new `synthesizeSpeech` in `src/lib/tts.ts`.
- Feature-flagged: `FEATURE_OUTBOUND_VOICE=false` by default. Tool/code path unreachable when off.
- New audit action `outbound_sent_voice` so deliveries are observable (incl. byte size for cost approximation).
- No regression: with the flag off, behaviour is byte-identical to pre-B4. With the flag on, the text-only path (B3a/B3b) is unchanged for non-voice inbound or long replies.
- Buffer-in-memory lifecycle: TTS bytes never touch disk (privacy + simplicity).

## 3. Non-goals

- **Voice replies for text inbound.** Strictly symmetric. Owner texted → Maia texts.
- **Voice for briefings, alerts, proactive worker messages.** Always text.
- **Voice cloning, premium voice quality, multi-voice support.** Voice hard-coded `'nova'` in B4.
- **`tts-1-hd` or `gpt-4o-mini-tts`.** `tts-1` is enough for short confirmations.
- **Cache by content-hash.** Replies contain dynamic values (saldos, IDs); hit rate is negligible. Regenerate every time.
- **Owner runtime toggle** of `FEATURE_OUTBOUND_VOICE` via WhatsApp ("para de me responder em áudio"). Post-B if requested.
- **Voice replies on turns that produced a PDF (B3b).** PDF wins; the document carries the substantive content with its caption.
- **Per-pessoa voice preference** (e.g., `pessoa.preferencias.voice_replies = false`). Single-user app today; YAGNI.
- **ElevenLabs / Azure / other providers.** OpenAI TTS only.
- **Streaming TTS.** B4 generates the full buffer before send.

## 4. Architecture

### 4.1 Decision tree (no-tool-uses branch in `core.ts`)

The existing branch order (after B3b) is: `latestReportPdf` → `usePoll` → text. B4 inserts a new branch **between** the PDF branch and the poll/text branches:

```typescript
if (latestReportPdf) {
  // existing B3b PDF branch (wins always)
} else if (
  config.FEATURE_OUTBOUND_VOICE &&
  inbound.tipo === 'audio' &&
  text.length <= OUTBOUND_VOICE_MAX_CHARS
) {
  // B4 voice branch (NEW)
} else if (usePoll && latestPending) {
  // existing B1 poll branch
} else {
  // existing B3a text branch (view-once + sendOutbound)
}
```

The two B4 guards (`inbound.tipo === 'audio'` and `text.length <= 400`) ensure precedence: PDF beats voice; voice beats poll/text **only** when both inbound was audio and reply is short.

**Why voice precedes poll**: a voice-in turn that also has a `latestPending` poll request is rare (would require the LLM to dispatch `ask_pending_question` after transcribing a voice question). When it happens, voice is the better UX (keeps the channel symmetry). The poll fallback to numbered text (B1) handles the case where Baileys can't honor the poll.

### 4.2 New module — `src/lib/tts.ts`

Mirrors `src/lib/whisper.ts`'s shape (single function, fetch-based, throws on failure):

```typescript
import { config } from '@/config/env.js';
import { logger } from '@/lib/logger.js';

export const OUTBOUND_VOICE_MAX_CHARS = 400;

/**
 * Synthesize Brazilian-Portuguese speech via OpenAI TTS. Returns the OGG-Opus
 * binary as a Buffer. Caller is responsible for delivering it via Baileys
 * (`sendOutboundVoice` in `gateway/baileys.ts`).
 *
 * Throws on HTTP error (caller falls back to text path).
 */
export async function synthesizeSpeech(text: string): Promise<Buffer> {
  if (!config.OPENAI_API_KEY) {
    throw new Error('OPENAI_API_KEY missing for TTS');
  }
  const t0 = Date.now();
  const res = await fetch('https://api.openai.com/v1/audio/speech', {
    method: 'POST',
    headers: {
      authorization: `Bearer ${config.OPENAI_API_KEY}`,
      'content-type': 'application/json',
    },
    body: JSON.stringify({
      model: 'tts-1',
      voice: 'nova',
      input: text,
      response_format: 'opus',
    }),
  });
  if (!res.ok) {
    const errText = await res.text();
    throw new Error(`tts_failed: ${res.status} ${errText}`);
  }
  const arrayBuffer = await res.arrayBuffer();
  const buf = Buffer.from(arrayBuffer);
  logger.debug({ ms: Date.now() - t0, bytes: buf.length }, 'tts.synthesized');
  return buf;
}
```

OpenAI's `response_format: 'opus'` returns Opus encoded inside an OGG container — exactly what WhatsApp/Baileys expects for voice notes. **No ffmpeg needed**, no external conversion step.

### 4.3 New `sendOutboundVoice` (`src/gateway/baileys.ts`)

Appended near `sendOutboundText` and `sendOutboundDocument`:

```typescript
/**
 * B4: send a voice note (PTT — push-to-talk). Renders as a voice-bubble in the
 * WhatsApp UI (the round one), not a music-player attachment. Buffer-only —
 * voice notes are small (≤100KB for ≤20s) and ephemeral; no disk persistence.
 */
export async function sendOutboundVoice(
  jid: string,
  buf: Buffer,
  opts?: { quoted?: WAQuotedContext },
): Promise<string | null> {
  if (!socket || !connected) {
    logger.warn('baileys.not_connected — cannot send voice');
    return null;
  }
  const result = await socket.sendMessage(
    jid,
    { audio: buf, mimetype: 'audio/ogg; codecs=opus', ptt: true },
    opts?.quoted ? { quoted: opts.quoted } : undefined,
  );
  return result?.key.id ?? null;
}
```

`ptt: true` is the Baileys flag that produces a voice-note bubble. Without it, the same audio renders as a music attachment with a player UI — wrong UX. `mimetype: 'audio/ogg; codecs=opus'` matches the format OpenAI TTS emits with `response_format: 'opus'`.

### 4.4 Agent-loop wiring (`src/agent/core.ts`)

Inside the no-tool-uses branch, add the voice path between the existing PDF branch (B3b) and the poll/text branches (B1/B3a):

```typescript
} else if (
  config.FEATURE_OUTBOUND_VOICE &&
  inbound.tipo === 'audio' &&
  text.length <= OUTBOUND_VOICE_MAX_CHARS
) {
  // B4: voice reply for symmetric voice-in/voice-out channel.
  let voiceBuf: Buffer | null = null;
  try {
    voiceBuf = await synthesizeSpeech(text);
  } catch (err) {
    logger.warn(
      { err: (err as Error).message, mensagem_id: inbound.id },
      'b4.tts_failed_fallback_text',
    );
  }
  if (voiceBuf) {
    const shouldQuote =
      (inbound.conteudo && detectCorrection(inbound.conteudo)) ||
      getActivePending(c) !== null;
    const wid = await sendOutboundVoice(jid, voiceBuf, {
      quoted: shouldQuote
        ? quotedReplyContext(
            inbound.metadata as Record<string, unknown> | null,
            inbound.conteudo,
          )
        : undefined,
    });
    if (wid) {
      await audit({
        acao: 'outbound_sent_voice',
        pessoa_id: pessoa.id,
        conversa_id: c.id,
        mensagem_id: inbound.id,
        metadata: {
          whatsapp_id: wid,
          char_count: text.length,
          byte_size: voiceBuf.length,
        },
      });
      await mensagensRepo.create({
        conversa_id: c.id,
        direcao: 'out',
        tipo: 'audio',
        conteudo: text,        // source text retained for audit/replay
        midia_url: null,        // ephemeral — never on disk
        metadata: {
          whatsapp_id: wid,
          remote_jid: jid,
          in_reply_to: inbound.id,
          voice: 'nova',
        },
        processada_em: new Date(),
        ferramentas_chamadas: [],
        tokens_usados: null,
      });
    }
  } else {
    // TTS failed — fall back to text path (re-uses existing sendOutbound).
    const shouldQuote =
      (inbound.conteudo && detectCorrection(inbound.conteudo)) ||
      getActivePending(c) !== null;
    await sendOutbound(pessoa.id, c.id, text, inbound.id, {
      pending_question_id: latestPending?.id ?? null,
      quoted: shouldQuote
        ? quotedReplyContext(
            inbound.metadata as Record<string, unknown> | null,
            inbound.conteudo,
          )
        : undefined,
    });
  }
}
```

Notes:

- **TTS failure → text fallback in the SAME branch.** We don't fall through to the next `else if` (poll) because that would skip the voice-in heuristic and might pick poll when text was fine. Owner gets a text reply matching what the LLM intended. View-once decisions don't apply (voice-in turns are rare to be sensitive saldos; cleaner to keep the text fallback simple).
- **Empty `text` is impossible here** because the outer branch already guards `if (text)`. The B4 branch lives inside that guard.
- **`OUTBOUND_VOICE_MAX_CHARS = 400`** is a constant in `src/lib/tts.ts`, imported by `core.ts`. Not env-configurable in B4 (YAGNI; if 400 is wrong, change the constant).

### 4.5 Storage / lifecycle

**Buffer in memory only.** TTS bytes (~10-100KB for ≤20s of audio) never touch disk. Sent directly via Baileys; the buffer is GC'd after `sendOutboundVoice` returns. Mirrors the privacy posture of B3b PDFs being deleted post-send and B3a view-once being client-side-only.

There is no cache. Replies contain dynamic figures (saldos, transaction IDs, entity names) — content-hash hit rate would be negligible. Regenerate every time at ~$0.001 per reply.

### 4.6 Branch precedence

The full no-tool-uses branch order after B4 lands:

1. `latestReportPdf` set → B3b PDF branch (with the LLM's text as caption). Wins always.
2. `FEATURE_OUTBOUND_VOICE=true` AND `inbound.tipo === 'audio'` AND `text.length ≤ 400` → B4 voice branch.
3. `usePoll && latestPending` → B1 poll branch.
4. Otherwise → B3a text branch (view-once + sendOutbound).

PDF beats everything; voice beats poll/text only when channel-symmetric and short. Poll beats text on multi-option pendings as before.

## 5. Schema / migrations

None. `mensagens.tipo` already accepts `'audio'` (it's used today by inbound voice notes). New audit action `outbound_sent_document` (sic: `outbound_sent_voice`) is appended to `src/governance/audit-actions.ts`.

## 6. Configuration

`FEATURE_OUTBOUND_VOICE` (default `false`) appended to `src/config/env.ts` after `FEATURE_PDF_REPORTS`.

When `false`:
- The B4 branch's first guard short-circuits; the path is unreachable.
- `synthesizeSpeech` is never called.
- No new HTTP traffic to OpenAI's TTS endpoint.
- Behaviour is byte-identical to pre-B4.

When `true`:
- B4 branch active. Owner sees voice replies on voice-in conversations with short replies.

Validation criteria for default-on flip:
- 5+ voice-in/voice-out turns observed in trial week.
- Owner reports voice-note quality "good enough" via single check-in.
- Average TTS latency observed < 2s (synthesizeSpeech to send) per reply.
- TTS failure rate < 1% over the trial.

`OUTBOUND_VOICE_MAX_CHARS = 400` is a constant in `src/lib/tts.ts`, NOT an env var. Rationale: if the cap needs adjusting, it's a code change with a test, not an operator knob.

## 7. Audit-action additions

In `src/governance/audit-actions.ts`:

```typescript
'outbound_sent_voice',
```

Single new action. The fall-back-to-text case reuses existing audit conventions (no special audit when TTS fails — the warn log captures it). The "skipped because >400 chars" case is invisible to audit — it just goes through the text path normally.

## 8. Concurrency

`synthesizeSpeech` makes one HTTP request per turn. Two concurrent voice-in turns produce two concurrent OpenAI TTS calls — no shared state. OpenAI TTS endpoint handles concurrency natively. No new locks, no cache invalidation, no shared resources.

## 9. Error handling

| Failure | Tratamento |
|---|---|
| `synthesizeSpeech` throws (network, 5xx, rate limit) | Log warn, fall back to text path inside the same branch (`sendOutbound`). Owner receives the reply as text. No `outbound_sent_voice` audit; existing `classification_suggested` audit (from the LLM turn dispatch) still fires as usual. |
| `OPENAI_API_KEY` missing | `synthesizeSpeech` throws `'OPENAI_API_KEY missing for TTS'`. Caught and treated like any TTS failure → text fallback. |
| `sendOutboundVoice` returns null (Baileys disconnected) | No audit, no `mensagens` row. Owner receives nothing — recovery worker (existing) requeues if appropriate. **No text fallback in this case** (text would also fail since Baileys is down). |
| `text.length === 0` | Already guarded by outer `if (text)` block. B4 branch is unreachable. |
| Reply contains emoji / special chars | OpenAI TTS handles Unicode gracefully (emoji become silent pauses). Acceptable. |
| Reply > 400 chars | Branch guard rejects; falls through to next branch (poll or text). No voice attempted. |
| Inbound was text but config flag is on | First guard rejects `inbound.tipo === 'audio'`. Falls through to text. |

## 10. Testing

### Unit (`tests/unit/tts.spec.ts` — new)

- `synthesizeSpeech` POSTs to `https://api.openai.com/v1/audio/speech` with the correct payload (`model: 'tts-1'`, `voice: 'nova'`, `response_format: 'opus'`, body `input: text`).
- Returns a Buffer when fetch succeeds.
- Throws when fetch returns non-2xx (`tts_failed: <status> <body>`).
- Throws when `OPENAI_API_KEY` is missing.
- `OUTBOUND_VOICE_MAX_CHARS` constant exported and equals 400.

### Unit (`tests/unit/baileys-send-voice.spec.ts` — new)

- Contract: `socket.sendMessage` receives `{ audio: Buffer, mimetype: 'audio/ogg; codecs=opus', ptt: true }`.
- `quoted` is forwarded as third arg when provided; `undefined` otherwise.
- Returns null when not connected; `sendMessage` not called.

### Unit (`tests/unit/voice-flow.spec.ts` — new)

- Voice-in + flag on + reply ≤400 chars → calls `sendOutboundVoice`; emits `outbound_sent_voice` audit; creates `mensagens` row with `tipo='audio'` and `midia_url=null`.
- Voice-in + flag on + reply >400 chars → falls through to text path; no voice call; no `outbound_sent_voice` audit.
- Text-in + flag on + reply ≤400 chars → text path; no voice call.
- Voice-in + flag off → text path; no voice call.
- Turn with `latestReportPdf` set → PDF branch wins regardless of inbound type or text length; voice never called.
- TTS failure (network throw) → text fallback fires; no `outbound_sent_voice` audit; warn log present.
- Baileys disconnect (null wid) → no audit, no `mensagens` row.

### Manual (PR checklist)

- Set `FEATURE_OUTBOUND_VOICE=true`. Send a voice note from the real Android receiver: "registra cinco reais do café".
- Verify Maia replies with a **voice-note bubble** (round, with PTT indicator), not a music-player widget.
- Verify the audio plays back correctly with `'nova'` voice in Brazilian Portuguese.
- Send a voice note that triggers a long reply (e.g., "manda o extrato do mês passado") — verify Maia generates the PDF (B3b) and DOES NOT produce a voice reply.
- Send a text message — verify Maia replies with text (no voice).
- With flag off: send a voice note → reply must be text. Confirm log shows no TTS HTTP call.

## 11. Out of scope

| Item | Defer to |
|---|---|
| Voice replies for text-in conversations | Not pursued (channel symmetry rule) |
| Voice for briefings / proactive worker messages | Not pursued (always text) |
| `tts-1-hd` or `gpt-4o-mini-tts` | Post-B if quality complaints arise |
| Voice cloning / ElevenLabs / Azure | Provider lock-in to OpenAI for B4 |
| Cache by content-hash | Hit rate negligible; not pursued |
| Owner runtime toggle of `FEATURE_OUTBOUND_VOICE` | Post-B |
| Per-pessoa voice preference | Single-user; YAGNI |
| Adjustable voice (male, neutral, etc.) | Not pursued (`'nova'` hard-coded) |
| Streaming TTS | Not pursued (full buffer is fine for ≤20s) |
| Voice replies for inbound > some threshold of seconds | Out of scope (the inbound length doesn't affect the voice/text choice — only the reply length does) |

## 12. Acceptance criteria

- [ ] `FEATURE_OUTBOUND_VOICE=false` → behaviour byte-identical to pre-B4. No new HTTP traffic to OpenAI TTS endpoint. The B4 branch is unreachable.
- [ ] `FEATURE_OUTBOUND_VOICE=true` + voice-inbound + reply ≤400 chars → `synthesizeSpeech` called once; `sendOutboundVoice` called with the resulting Buffer + `ptt: true`; audit `outbound_sent_voice` fires with `whatsapp_id`, `char_count`, `byte_size`; outbound `mensagens` row created with `tipo='audio'` and `midia_url=null`.
- [ ] `FEATURE_OUTBOUND_VOICE=true` + voice-inbound + reply >400 chars → falls through to text path; no voice call; no `outbound_sent_voice` audit. The text path's view-once decision (B3a) fires normally.
- [ ] `FEATURE_OUTBOUND_VOICE=true` + text-inbound → text path always; no voice call (regardless of reply length).
- [ ] `latestReportPdf` set on a voice-inbound turn → PDF branch wins; voice never dispatched.
- [ ] TTS failure (`synthesizeSpeech` throws) → text fallback inside the same branch (`sendOutbound` called); no `outbound_sent_voice` audit; warn log emitted.
- [ ] Baileys disconnected (`sendOutboundVoice` returns null) → no audit, no `mensagens` row. Owner receives nothing (existing recovery handles).
- [ ] Voice note renders as a **voice-note bubble** in WhatsApp UI (PTT — round bubble, not rectangular music player).
- [ ] Pre-existing tests do not regress: B0 (pending-gate), B1 (one-tap), B2 (message-update + reminder), B3a (view-once), B3b (PDF flow).
- [ ] `FEATURE_OUTBOUND_VOICE` documented in `.env.example`.
